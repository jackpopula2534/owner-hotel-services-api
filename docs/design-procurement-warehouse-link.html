01

ปัญหาปัจจุบัน & Gaps

วันนี้เมื่อ approve PO แล้ว ระบบสร้าง รายการรอรับเข้า ในฝั่งคลัง แต่ฝั่งจัดซื้อยัง ไม่เห็น ว่ารับครบแล้วหรือยัง, ของในคลังเหลือเท่าไหร่, ใกล้หมดอายุชิ้นไหน — ต้องเดินไปถามคลัง

Procurement (ปัจจุบัน)

มีอยู่
  • สร้าง PR / RFQ / PO / เทียบราคา
  • เห็นสถานะ PO: DRAFT / PENDING / APPROVED ฯลฯ
  • มี GR list แต่ไม่ผูกกลับมาแสดงในมุม PO

Warehouse (ปัจจุบัน)

มีอยู่
  • WarehouseStock, StockMovement, InventoryLot
  • GR + QC + Lot/Expiry tracking
  • Reorder alerts page (ภายในคลัง)

Gaps ที่ต้องเชื่อม

  1. PO ไม่ได้แสดง receivedQty / pendingQty ระดับบรรทัด
  2. ไม่มีหน้า "PO ที่รอรับเข้า" รวมศูนย์
  3. ฝั่งจัดซื้อไม่เห็น Stock balance + ของใกล้หมด
  4. ไม่มี PO Aging (รอนานกี่วัน, เกินกำหนด)
  5. ไม่มี variance report (สั่ง vs รับ vs ยอด invoice)
  6. ไม่มี event push เวลา GR เสร็จ → ฝั่ง PO refresh เอง
  7. ของ ใกล้หมดอายุ / หมดแล้ว ไม่โผล่ฝั่งจัดซื้อให้ตัดสินใจซื้อใหม่/คืน
  8. Overstock ไม่มีสัญญาณเตือนกลับฝั่งจัดซื้อ → ซื้อซ้ำซ้อน

03

State Machine — PO Lifecycle

เพิ่มเฉพาะ trigger ที่มาจาก GR (เส้นประสีเขียว) — รัฐคงเดิมตาม Prisma enum

stateDiagram-v2 [*] --> DRAFT DRAFT --> PENDING_APPROVAL: submit PENDING_APPROVAL --> APPROVED: approve PENDING_APPROVAL --> DRAFT: reject (with reason) APPROVED --> PARTIALLY_RECEIVED: GR.complete (qty < ordered) APPROVED --> FULLY_RECEIVED: GR.complete (qty = ordered) PARTIALLY_RECEIVED --> FULLY_RECEIVED: GR.complete (cumulative = ordered) PARTIALLY_RECEIVED --> CLOSED: force close (with variance reason) FULLY_RECEIVED --> CLOSED: invoice matched + paid APPROVED --> CANCELLED: cancel before any GR PENDING_APPROVAL --> CANCELLED: cancel CANCELLED --> [*] CLOSED --> [*]
Trigger จาก GR
เมื่อ GR ผ่าน QC (status=ACCEPTED) ระบบจะ recalculate receivedQty ของแต่ละ PO line แล้ว set status PO อัตโนมัติ
Force Close
ถ้าซัพไม่ส่งครบและไม่จะรับเพิ่ม → ปิด PO (ระบุเหตุผล) แล้วยอดส่วนต่างจะปรากฏใน Variance Report
Cancel
อนุญาตยกเลิกได้ก่อนมี GR เท่านั้น — ถ้ามี GR แล้วต้องใช้ Force Close

Event & Notification Design

เชื่อมแบบ event-driven ผ่าน Bull queue + Socket.IO + push notification

sequenceDiagram autonumber participant U as ผู้ใช้คลัง participant GR as GoodsReceiveService participant DB as Prisma participant BUS as EventBus (Bull) participant PO as PurchaseOrderService participant WS as Socket.IO participant N as NotificationService participant UI as ฝั่งจัดซื้อ U->>GR: complete inspection (ACCEPTED) GR->>DB: write GoodsReceiveItem + StockMovement + Lot GR->>BUS: emit gr.completed { poId, items[] } BUS->>PO: handler — recalc receivedQty PO->>DB: update PurchaseOrder.status (PARTIAL/FULL) PO->>BUS: emit po.received { poId, status } BUS->>WS: broadcast room:procurement BUS->>N: queue toast/email/LINE WS-->>UI: live update PO Tracking card N-->>UI: 🔔 inbox + push
Event
gr.completed
payload: { grId, poId, items: [{ poItemId, qty, lotId, expiry }] }
consumer: PurchaseOrderService.recalcReceived, NotificationService
Event
po.received
payload: { poId, oldStatus, newStatus, percent }
consumer: WS broadcast → procurement room
Event
stock.low
trigger: WarehouseStock.quantity ≤ item.reorderPoint (post-write hook)
consumer: dashboard widget refresh + นำเสนอใน Stock Report tab "ใกล้หมด"
Event
lot.expiring
trigger: cron daily — find lots where expiryDate ≤ today + N days
consumer: ส่งเข้า inbox จัดซื้อ + ทำ flag ใน Stock Report
Event
po.overdue
trigger: cron — PO APPROVED ที่ expectedDate < today และ status ≠ FULL
consumer: highlight ใน Kanban "เกินกำหนด" + แจ้ง buyer + email supplier
Channel
การแจ้งเตือน
  • Toast (real-time) — Socket.IO
  • Inbox bell — persistent
  • Email — daily digest
  • LINE Notify — สำหรับ overdue/expired

GET /api/v1/inventory/procurement-stock/balance ?warehouseId=&filter=LOW|OVERSTOCK|NEAR_EXPIRY|EXPIRED|ALL &daysToExpiry=30&page=1&limit=50 → { data: [{ itemId, sku, name, qty, reorderPoint, minStock, maxStock, status: 'LOW'|'OK'|'OVER', supplier: { id, name, leadDays }, openPo: { poId, poNumber, expectedDate } | null }] } GET /api/v1/inventory/procurement-stock/expiring ?days=30&warehouseId= → [{ lotId, itemId, sku, qty, expiryDate, daysLeft, value, status: 'NEAR'|'EXPIRED' }] GET /api/v1/inventory/procurement-stock/summary → { lowCount, overstockCount, expiringSoonCount, expiredCount, stockValue, monthRcvValue } POST /api/v1/inventory/procurement-stock/create-pr-from-low body: { itemIds: [...], warehouseId, requestedBy } → 201 { prId, prNumber, items[] } // สร้าง PR จากของใกล้หมดที่เลือก
GET /api/v1/inventory/reports/po-variance ?from=2026-04-01&to=2026-04-30 → [{ poId, poNumber, ordered, received, invoiced, deltaQty, deltaAmount, reason, suggestedAction }] POST /api/v1/inventory/notifications/subscribe body: { events: ['gr.completed','po.received','po.overdue', 'stock.low','lot.expiring'] } // socket.io room: 'procurement:{tenantId}'

Schema Additions (ขั้นต่ำ)

model PurchaseOrder { // ...existing fields expectedDeliveryDate DateTime? forceClosedAt DateTime? forceClosedReason String? varianceQty Int? // คำนวณ trigger varianceAmount Decimal? // คำนวณ trigger } model PurchaseOrderItem { // ...existing fields receivedQty Int @default(0) // rollup จาก GR pendingQty Int @default(0) // generated lastReceivedAt DateTime? } model ProcurementSubscription { // ใหม่ id String @id @default(cuid()) userId String events String[] // gr.completed, po.overdue, ... channel String // SOCKET | EMAIL | LINE active Boolean @default(true) createdAt DateTime @default(now()) }

Cases Reference — ที่คุณถามมา

Case 1
PO รอรับเข้า
เห็นที่ PO Tracking → คอลัมน์ Approved · จำนวน & มูลค่ารวมโผล่บน Dashboard KPI
Case 2
PO รับเข้าแล้ว
คอลัมน์ FULL + ปุ่ม PDF / Close
Case 3
PO รับเข้ายังไม่หมด (รับครึ่ง)
คอลัมน์ PARTIAL + progress bar + ตาราง line-level สั่ง / รับ / ค้าง ใน PO Detail
Case 4
PO เกินกำหนด
Cron ทุกเที่ยงคืน · เปลี่ยนสีบน Kanban + แจ้งเตือน buyer + อีเมลซัพ
Case 5
ของใกล้หมด (Low Stock)
Stock Report tab "⚠ ใกล้หมด" + เลือกหลายรายการ → "+ สร้าง PR" รวบ batch
Case 6
ของเยอะเกิน (Overstock)
Tab "📈 Overstock" — ระบบ บล็อกการสร้าง PR/PO ของ SKU นั้นจนกว่าจะลด · มี action "คืนซัพ"
Case 7
ของใกล้หมดอายุ
Tab "⏳ ใกล้หมดอายุ" + lot-level + มูลค่ารวม · suggest action: promo / FIFO / RETURN_SUPPLIER
Case 8
ของหมดอายุแล้ว
Tab "⛔ หมดอายุแล้ว" — บันทึก WASTE เข้า StockMovement + เคลม supplier
Case 9
เห็น GR ที่คลังบันทึก realtime
Event gr.completed → Socket.IO push เข้า room procurement → card อัปเดต progress bar
Case 10
PO ↔ Invoice ไม่ตรง
Variance Report → 3-way match (PO ↔ GR ↔ Invoice) + ปุ่ม "โต้แย้ง invoice"
